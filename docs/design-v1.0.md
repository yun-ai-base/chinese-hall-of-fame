## 一、产品概述

### 1.1 核心理念

以太阳系为隐喻框架，将中华五千年文明史可视化为一个可探索的交互星系。太阳代表中华文明，八大行星代表八个文化维度，每颗行星的卫星代表该维度的核心名人（共 361 位），名人的关联人物构成更深一层轨道。

### 1.2 三层星系结构

```
第0层 (恒星)        中华文明
                      │
第1层 (行星)    八大文化维度 (361位名人)
                      │
第2层 (卫星)    各维度核心名人
                      │
第3层 (子卫星)  名人的关联人物 (师承/同僚/论敌/传人)
```

### 1.3 交互深度规则

- **层级锁定**：默认只允许 3 层深度导航（维度 → 名人 → 关联人物）
- **第 3 层限制**：点击关联人物——
  - 若该人物属于在册名人 → 直接跃迁到对应维度的名人视图
  - 若为边缘人物（不在在册名单内）→ 弹出信息面板，不跃迁
- **第 3 层视觉区分**：关联人物星球缩小 40% + 亮度降低，hover 提示"查看详情"
- **深度突破**：信息面板内提供"以此人为中心重构星系"按钮，主动选择后方可突破层级限制
- **返回机制**：导航历史栈支持逐级返回，始终有"返回上一级"入口
- **历史栈上限**：20 条，超出时丢弃最早条目

### 1.4 太阳（中华）交互

- 点击太阳 → 弹出文明总览面板（时间轴 + 维度导览）
- 面板底部设"随机探索"按钮 → 随机跳转一位名人

## 三、数据结构设计

### 3.1 TypeScript 接口定义

```typescript
// 文化维度
interface Dimension {
  id: string;
  name: string;
  nameEn: string;
  planetIndex: number;
  color: string;
  description: string;
  categories: Category[];
}

interface Category {
  name: string;
  start: number;
  end: number;
}

// 名人
interface Figure {
  id: string;
  name: string;
  styleName?: string;          // 字号/别称，如"字太白，号青莲居士"
  pinyin: string;
  dynasty: string;
  era: EraRange;
  eraLabel?: string;
  gender?: '男' | '女';
  honor?: string;
  dimension: string;
  dimensionCategory: string;
  crossDimension?: string[];
  tags: string[];
  summary: string;
  avatar: string;
  birthplace?: string;
  deathplace?: string;
  locations?: string[];
  geo?: Record<string, [number, number]>;  // 地点坐标，如 {"长安":[108.94,34.26]}
  relations?: Relation[];
  details?: FigureDetails;
}

interface EraRange {
  start: number | null;
  end: number | null;
  approximate: boolean;
}

interface Relation {
  targetId: string;
  relation: string;
  reverseRelation?: string;
  description?: string;
}

interface FigureDetails {
  bio: string;
  ideology?: string;
  group?: string;
  timeline: TimelineEvent[];
  works?: Work[];               // 有就填。文学人物可含 fullText 完整诗文
  achievements?: string[];
  quotes?: string[];
  appraisals: string[];
  influence: string;
  controversies?: string[];
  mediaPortrayals?: string[];
  memorials?: string[];
  gallery: GalleryItem[];
  trivia: string[];
  references?: string[];
}

interface TimelineEvent {
  year: string;
  event: string;
  type: 'birth' | 'death' | 'milestone' | 'work';
}

interface Work {
  title: string;
  year?: string;
  type: string;
  summary?: string;
  fullText?: string;            // 完整诗文选录，仅供诗词文学维度代表作品
  image?: string;
}

interface GalleryItem {
  src: string;
  title: string;
  source?: string;
  license?: string;
}
```

### 3.2 双向关系自动映射

`relations` 为唯一数据源，`associated` ID 列表由代码自动提取。运行时构建邻接表自动补全双向边。

### 3.3 神话传说的分类边界说明

在"神话传说"维度中，涵盖三类不同来源的人物，UI 层面通过分类标签明确区分：

- **上古创世·始祖**：原始信仰/神话体系，无确切历史记载
- **山海经·上古神话**：山海经记载的神怪传说
- **神魔小说**：明清小说（封神/西游）的文学虚构人物
- **民间传说**：口口相传的民间故事人物

分类标签在子轨道和详情面板中始终可见，避免文化科普上的歧义。

### 3.4 数据维度说明（中文分类）

每条名人数据包含以下维度：

**基本信息**：人物ID、姓名、拼音、朝代、生卒年、时期标签、性别、后世名号、一句话简介

**归属信息**：所属维度、所属子分类、交叉维度、标签

**关联人物**：关联人物ID、关系类型、反向关系、关系说明

**生平年表**：年份标识、事件描述、类型

**成就与作品**：代表作、核心成就、经典语录

**核心理念**：思想主张一句话概括（哲学类人物关键字段）

**交游圈子**：群体归属，如竹林七贤、唐宋八大家

**历史评价**：后人对该人物的定论

**历史影响**：对后世的传承脉络

**相关争议**：争议性人物的正反观点

**主要地点**：出生地、卒地、活动地

**纪念地**：故居、祠堂、纪念馆

**影视形象**：影视文学作品呈现

**冷知识**：趣闻轶事

**图库**：公共领域作品图片

**参考资料**：参考史料与延伸阅读

### 3.6 两类关联人物

点击名人跃迁成为新中心后，环绕他的关联人物分为两类，视觉和行为均有区分：

**在册关联人物（360 人名单内）：**
- 星球颜色：该人物所属维度的维度色
- 交互：可点击跃迁，直接跳到该人物的名人视图
- 数据：拥有完整的 18 个维度

**非在册关联人物（360 人名单外）：**
- 星球颜色：灰色半透明
- 交互：不可跃迁，点击仅弹出基础信息面板
- 数据结构精简：

```typescript
interface AssociatedPerson {
  id: string;
  name: string;
  summary: string;             // 一句话简介
  relation: string;            // 与当前名人的关系
  relationDesc?: string;       // 关系详细说明
  works?: Work[];              // 有就填，没有则不填
  baiduBaike: string;          // 百度百科词条链接（唯一跳转来源）
}
```

> 注：百度百科链接仅用于非在册关联人物，在册名人以内链数据为准、不跳转外部页面。

不存朝代、生卒年、年表等完整维度，体积小、加载快。

### 4.3 渲染技术

- **主材质**：Canvas 水墨纹理贴图
- **光晕**：Sprite 图层 + 径向渐变
- **太阳**：自定义 Shader（表面流动光效 + Fresnel 辉光），移动端简化版
- **星球名称标签**：白色文字 + 维度色 text-shadow 发光，无 pill 背景
- **轨道线**：维度色 LineBasicMaterial，透明度 0.12

### 4.4 字体

## 五、交互规范

### 5.1 交互状态机

```
[宇宙视图] ──点击维度──> [维度视图] ──点击名人──> [名人视图] ──点击关联──> [面板弹出]
     ^                     ^                       ^                      │
     └──返回──┘       └──返回──┘             └──返回──┘         [点"重构"]→跃迁
```

### 5.2 转场动画

- **跃迁**：桌面端 1.5s / 移动端 1.0s，easeInOutCubic
- **面板**：0.3s 弹性滑入
- **载荷态**：移动端跃迁期间显示脉冲圆点，数据就绪后生成星球
- **子轨道分层布局**：名人超过 20 人的维度，卫星按子分类分配到多个半径轨道层（每层间距 2，每层 10-15 人）。同一轨道的人物按年代沿圆周顺序排布，年代越早越靠近行星（内轨），越晚越靠外，在空间布局中承载时间轴线信息
- **自动回拢**：用户停止操作 15 秒后，相机缓慢拉远至全局视角，并自动对齐黄道面
- **面包屑导航**：顶部常驻显示当前路径（如「中华文明 → 诗词文学 → 李白」），点击可快速跳回任意层级

### 6.2 无构建工具策略

- Three.js 文件下载到 `vendor/three/`，通过 importmap 指向本地路径
- 无需 npm build 步骤，纯静态文件
- **本地开发必须用 HTTP 服务器**（`python -m http.server 8000`），不支持 `file://` 协议
- **字体优化**：Noto Serif/Sans SC 使用 `font-display: swap`，上线前对字体做子集化（提取 3500 常用字 + 在册姓名用字），将体积从十几 MB 压缩至 1-2 MB

### 6.4 数据加载策略（两级加载）

- **维度级**：点击维度时，一次性加载该维度所有名人的基础信息（姓名/朝代/简介/ID），用于卫星渲染
- **人物级**：点击具体名人时，按需加载该人物的完整详情数据（年表/作品/图库）
- **LRU 缓存**：保留最近访问的 2~3 个层级场景资源，超出上限时 dispose 最早访问的资源，提升往返切换流畅度

### 6.5 移动端降级

- 星尘粒子数量：800（桌面 3000）
- 太阳 Shader：简化 flow 层 + 关闭 Fresnel
- 相机默认俯角加大（2.5D 视角）
- 跃迁时长：1.0s
- 渲染分辨率限制：pixelRatio ≤ 1.5
- **维度视图预览模式**：进入维度后默认只展示子分类标签，点击分类才展开该分类的全部卫星，避免 60 人同屏拥挤误触
- **远景标签自动隐藏**：相机拉远时自动隐藏卫星名称标签，拉近到一定距离再显示，保持远景画面干净
- **触控优化**：点击热区比星球视觉尺寸大 50%，提升小屏幕点击精度
- **面板布局**：信息面板默认半屏展示，上滑展开全屏，避免遮挡星系内容
- **滑动返回手势**：适配移动端原生操作习惯

### 6.6 dispose 协议

```javascript
disposeEntity(object) {
  if (object.geometry) object.geometry.dispose();
  if (object.material) {
    if (object.material.map) object.material.map.dispose();
    if (object.material.emissiveMap) object.material.emissiveMap.dispose();
    object.material.dispose();
  }
  if (object.parent) object.parent.remove(object);
}
```

## 八、设计原则

1. **数据驱动**：所有展示内容来自 JSON，渲染层不硬编码任何名人信息
2. **离线可用**：天体纹理用 Canvas 动态生成；面板内容图走 Wikimedia Commons 公共领域图源
3. **渐进增强**：桌面端全 3D 体验，移动端降级但不丢功能
4. **单向数据维护**：关联关系只写一次，前端自动双向补全
5. **内容 > 形式**：动画和视觉效果服务于内容展示，不喧宾夺主
6. **资源即用即释**：每次跃迁必须完整 dispose 上一级天体资源（geometry / material / texture）
7. **图片来源规范**：面板内容图使用公共领域图源，gallery 字段标注 source 与 license
